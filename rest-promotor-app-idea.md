# 休憩プロモーター — プロジェクト概要ドキュメント

> Claude Code での実装引き継ぎ用。ここまでの設計・議論・決定事項をすべて記録する。

---

## 1. プロダクトビジョン

過集中しやすいユーザーが、自己判断を損なわずに適切な休憩を取れるよう促すローカル Web アプリケーション。  
「強制」ではなく **「自己認識の鏡」** を提供することで、持続可能な作業リズムを支援する。

### 背景・課題

- 過集中の性質があり、集中すると休憩をないがしろにして疲労困憊しやすい
- 通知・タイマーは「無視して続ける」が容易で効果が薄い
- 自分の疲労や状態の悪化に気づきにくい

### 解決アプローチ

単なるリマインダーではなく、**介入時に会話UIを出してユーザーに内省を促す**。  
返答する過程で自分の状態を言語化させることで、無視しにくくする。  
最終的な判断は常にユーザーに委ねる（強制ではなく自律的な選択）。

## 2. 配布形式・技術スタック

| 項目 | 決定内容 |
|------|----------|
| 形式 | ローカル Web アプリ（自分用、配布不要） |
| サーバー | FastAPI + uvicorn |
| UI | HTML / CSS / JS（既存プロトタイプを流用・表示専用に絞る） |
| ロジック | タイマー判定・スコア処理・ログ書き込みを Python 側に集約 |
| ログ永続化 | CSV ファイル（pandas で分析可能） |
| 起動方法 | `.bat` ファイルをダブルクリック → サーバー起動 → ブラウザ自動オープン |
| パッケージ管理 | uv |
| 配布 | フォルダをそのまま使う（exe 化不要） |
| OS | Windows |

### 依存パッケージ（最低限）

```toml
# pyproject.toml
[project]
dependencies = [
    "fastapi",
    "uvicorn[standard]",
    "pandas",
]
```

### アーキテクチャ方針

```
[ブラウザ UI]
    │  HTTP ポーリング（1秒）
    ▼
[FastAPI サーバー（Python）]
    ├── routers/      # エンドポイント定義のみ
    ├── services/     # ビジネスロジック（タイマー管理・モード判定・アドバイス分岐）
    └── data/         # CSV 読み書き
```

- **UI はレンダリングと入力受付のみ**。ビジネスロジックをブラウザに持たせない
- ロジックは `routers/` ではなく **`services/` 層に集約**する。フェーズ2でのAI会話追加を見越した分離
- ログが CSV に永続化されるためページリロードしてもデータが消えない

## 3. 主要機能（MVP フェーズ1）

### 3-1. タイマー

- デフォルト **60分** のカウントダウン
- 操作：開始・一時停止・リセット
- タイマー 0 到達時に介入ダイアログを自動表示
- 休憩中はセッション経過時間のカウントを停止

### 3-2. 累積作業時間の管理

2 種類の時間を **分離して管理** する。

| 変数 | 意味 | リセットタイミング |
|------|------|--------------------|
| セッション経過時間（`session_elapsed`） | 現在のタイマーセッションで作業した時間 | スヌーズ時・休憩時ともにリセット |
| 累積作業時間（`accum_elapsed`） | 最後に休憩してから一切休憩なしで作業した合計時間 | **休憩を取ったときのみリセット**。スヌーズでは継続 |

累積作業時間の閾値：

| 累積時間 | 状態 | UI 表示 |
|----------|------|---------|
| ～134分 | 通常 | 累積バー：通常色 |
| 135～179分 | 警告 | 累積バー：黄色 |
| 180分以上 | 上限 | 累積バー：赤色 |

### 3-3. 強制介入ダイアログ

タイマー発火時にオーバーレイで表示。**無視不可**（別操作をしない限り閉じない）。

ダイアログは 4 つのモードに分岐する：

| モード | 発動条件 | 特徴 |
|--------|----------|------|
| `first`（初回） | アプリ起動後の最初の開始、またはリセット後の再開始 | 「これからどんな作業？」から始まる。休憩/スヌーズ選択なし |
| `timer`（タイマー介入） | タイマー発火時（`accum_elapsed` が上限未満） | 今の状態確認 → 作業内容（前回値デフォルト）→ 休憩 or スヌーズ選択 |
| `self`（自己申告） | 自己申告ボタン押下時（`accum_elapsed` の状態を問わず） | `timer` と同フロー。冒頭文言のみ「自己申告：」に変更 |
| `force`（強制休憩） | タイマー発火 + `accum_elapsed` が上限以上 | 警告バナー表示・スヌーズ選択肢を非表示・休憩15分固定 |

**`force` と自己申告の重複について**：  
`accum_elapsed` が上限超過中に自己申告ボタンを押した場合は `self` モードで開く。ユーザーの意思を尊重し、休憩・スヌーズともに選択可能。

**休憩終了後の挙動**：  
休憩タイマーが 0 になったら、サーバー側が `dialog_triggered = true` かつ `dialog_mode = "timer"` をセットし、UI 側が自動で `timer` フローのダイアログを起動する。ユーザーの操作なしに次の作業セッションが始まる。

### 3-4. ダイアログで収集する情報

各介入時に以下を記録する：

| 項目 | 形式 | 備考 |
|------|------|------|
| 作業内容 | テキスト入力 | 前回値をデフォルト表示。未入力でステップ進行不可（バリデーションエラー表示） |
| 作業負荷 | 1〜5の5段階選択 | 1=軽い、3=普通、5=重い。前回値をデフォルト選択 |
| 状態スコア | 1〜5の5段階選択 | 1=かなり悪い、5=かなりよい |
| 判断 | 休憩（5/10/15分）またはスヌーズ（15/30/45/60分） | `force`時は休憩15分固定 |

`first` モードで記録した内容（作業内容・負荷・状態スコア）も、グラフとログに反映する。

**作業負荷の目安（参考）**：

| 負荷レベル | 作業例 |
|-----------|--------|
| 1（軽い） | 日報作成、メール確認 |
| 3（普通） | 資料作成、ミーティング |
| 4〜5（重い） | コーディング、論文読解 |

### 3-5. 状態スコアに基づくアドバイス分岐

アドバイス判定には `session_elapsed`（現セッション経過時間）を使用する。`accum_elapsed` とは別物なので注意。

| 条件 | アドバイス・UI変化 |
|------|-------------------|
| 状態スコア 1〜2 | ダイアログ枠が赤ボーダー。強い警告バナー表示 |
| 状態スコア 3 かつ `session_elapsed` が 90分超 | 警告バナー表示。休憩を勧める |
| 負荷 4〜5 かつ `session_elapsed` が 45分超 | 警告バナー表示。認知負荷の蓄積を説明 |
| その他 | 状態良好として継続を支持 |

### 3-6. 表示グラフ（メイン画面）

| グラフ | 種類 | 内容 |
|--------|------|------|
| 状態スコア推移 | 折れ線 | 記録ごとのスコア推移。点の色：1〜2=赤、3=グレー、4〜5=緑 |
| 連続作業時間 | 棒グラフ | 各セッションの連続作業時間（分） |

ダイアログ内にも状態スコア推移グラフを表示（「今回」を追加した状態でプレビュー）。

### 3-7. ログ表示

- メイン画面に直近8件を表示
- 各レコード：時刻・作業内容・負荷（ドット表示）・状態スコア（色付きラベル）・行動（休憩/継続）
- **CSV に永続化**（pandas で分析可能）

### 3-8. 自己申告ボタン

- 画面常駐の小さいボタン
- 押すと `self` モードのダイアログを起動
- タスクの区切りなどユーザーが自律的に休憩を申告するための導線

## 5. UX設計の原則

### 「無視しにくい」設計

- 通知ではなく **オーバーレイで全画面をブロック**
- 問いかけへの返答という行動コストを発生させる
- 返答しながら自分の状態を言語化 → 内省が起きる
- 内省した結果として自律的に判断させる（強制ではない）

### 自律性の確保

- アドバイスや警告を出しても、最終判断は常にユーザー
- スコアが低くても「作業を続ける」は選択可能
- `force` モードのみ例外として休憩を強制（180分は健康上の上限として設定）

### 情報設計

介入時にユーザーが判断に使える情報を3軸で提示する：

1. **状態スコア推移グラフ**（過去から今回まで）
2. **連続作業時間**（今どのくらい連続しているか）
3. **作業負荷**（今やっていることの認知的重さ）

## 6. 受け入れ条件（MVP）

- タイマー発火時に画面を覆うオーバーレイが表示され、別の操作をしない限り閉じない
- 状態スコアが 1〜2 の場合、ダイアログ枠に警告色（オレンジ〜赤）が表示される
- 累積作業時間が 135分 に達したとき累積バーが黄色になる
- 累積作業時間が 180分 に達したとき累積バーが赤になる
- `force` モードはタイマー発火時のみ発動する（タイマー計測中のリアルタイム割り込みはしない）
- `force` モード時はスヌーズ選択肢が非表示になり、休憩時間は 15分 固定となる
- `accum_elapsed` が上限超過中に自己申告ボタンを押した場合は `self` モードで開く
- 休憩を実際に選択したときのみ、`accum_elapsed` がゼロにリセットされる
- `first` モードはアプリ起動後の最初の開始、およびリセット後の再開始時に発動する
- `first` モードで記録した内容はグラフとログに反映される
- 前回の作業内容・負荷がデフォルト値として表示される
- 作業内容テキストが未入力の状態で次へ進もうとした場合、エラー表示してステップを進めない
- ログが CSV ファイルに書き込まれ、アプリ再起動後も参照できる
- 休憩タイマーが 0 になったら自動で `timer` フローのダイアログを起動する

## 8. API 設計

| エンドポイント | メソッド | 概要 |
|----------------|----------|------|
| `GET /state` | GET | タイマー残り秒・`session_elapsed`・`accum_elapsed`・`is_running`・`is_breaking`・`dialog_triggered`・`dialog_mode`・`prev_task`・`prev_load` を返す |

| `GET /logs` | GET | ログ一覧を返す（グラフ・ログ表示用） |

## 10. タイマー実装の設計方針（重要）

### 方針

タイマーは **Python サーバー側のバックグラウンドスレッドで動かす**。UI（JS）は表示を同期するだけ。

- UI（JS）側で動かす案はシンプルだが、ページリロードやタブ非アクティブで状態が失われる
- サーバー側で動かすことで状態がサーバーに集約され、リロードしても継続できる

### TimerState の構造

```python
# services/timer_service.py
class TimerState:
    is_running: bool          # タイマー動作中フラグ
    is_breaking: bool         # 休憩中フラグ（True のとき session_elapsed をカウントしない）
    is_first: bool            # first モードフラグ（起動時・リセット後に True）
    remaining: int            # カウントダウン残り秒
    session_elapsed: int      # 現セッション経過秒（スヌーズ・休憩でリセット）
    accum_elapsed: int        # 累積作業秒（休憩時のみリセット）
    dialog_triggered: bool    # ダイアログ発火フラグ（UI がポーリングで検知）
    dialog_mode: str | None   # 発火するダイアログのモード
    prev_task: str            # 前回の作業内容（次回ダイアログのデフォルト値）
    prev_load: int            # 前回の作業負荷（次回ダイアログのデフォルト値）
```

バックグラウンドスレッドは `threading.Thread` で実装し、1秒ごとに状態を更新する。

### `GET /state` のレスポンス例

```json
{
  "remaining": 3542,
  "session_elapsed": 58,
  "accum_elapsed": 4258,
  "is_running": true,
  "is_breaking": false,
  "dialog_triggered": false,
  "dialog_mode": null,
  "prev_task": "論文読解",
  "prev_load": 5
}
```

### ダイアログ発火フローの詳細

```
[バックグラウンドスレッド]
remaining が 0 になる
  → ダイアログモードを判定（timer / force）
  → dialog_triggered = true, dialog_mode = "timer" or "force" をセット
  → タイマーを停止

[UI（JS）- 1秒ポーリング]
GET /state の response.dialog_triggered が true を検知
  → POST /dialog/ack を叩く（二重発火防止）
  → dialog_mode に応じたダイアログを表示

[ユーザーが判断を送信]
POST /record を叩く
  → サーバー側でログ書き込み・状態更新・次タイマーセット
  → タイマー再起動
```

### ポーリング vs WebSocket

| 方式 | メリット | デメリット |
|------|----------|------------|
| ポーリング（1秒）| 実装シンプル、デバッグしやすい | 最大1秒の表示ラグ |
| WebSocket | リアルタイム、サーバーからプッシュ可能 | 実装複雑度が上がる |

**MVP はポーリングで実装する。** タイマー表示の最大1秒ラグは許容範囲とする。WebSocket への移行はフェーズ2で検討する。
