## DEC-001 タイマー開始タイミング（`is_first` 時は `POST /record` まで待つ）

**日付:** 2026-05-09

**背景:**
`POST /start` を呼んだ時点で `is_running = True` にすると、`first` モードのダイアログを操作している間もタイマーがカウントダウンしてしまう。

**選択肢:**
- A. `POST /start` 時点で即 `is_running = True`（シンプルだが入力中にカウントが進む）
- B. `is_first` のとき `dialog_triggered = True` のみセット。`POST /record action="start"` 完了後に `is_running = True`（ダイアログ入力中はタイマー停止）

**決定:** B を採用

**理由:**
ユーザーが作業内容・負荷・状態スコアを入力している時間は作業時間ではない。入力完了後にタイマーを開始するのが正確な計測につながる。

**docs/ への反映:** 反映済み（`functional-design.md` の `POST /start` 仕様）

## DEC-002 `first` モードの `action` 値に `"start"` を定義

**日付:** 2026-05-09

**背景:**
CSV の `action` 列に `first` モード完了時の値が未定義だった。`"rest"` / `"skip"` しか仕様になかった。

**選択肢:**
- A. `first` モードは `action` 列を空欄にする
- B. `"start"` を新しい `action` 値として定義する

**決定:** B を採用

**理由:**
空欄だとログ集計時の扱いが曖昧になる。`"start"` とすることで「セッション開始」として意味が明確になり、グラフの `first` レコード除外判定（`session_min = 0`）と二重の識別手段を持てる。

**docs/ への反映:** 反映済み（`functional-design.md` の CSV データモデル）

## DEC-003 `POST /advice` エンドポイントの追加

**日付:** 2026-05-09

**背景:**
`session_service.get_advice()` をダイアログ Step1 完了後に呼ぶ手段がなかった。`POST /record` は最終ステップ送信時に呼ぶため、途中ステップでアドバイスを取得できない。

**選択肢:**
- A. `GET /state` のレスポンスにアドバイスを含める（常時計算・常時返却）
- B. `POST /advice` として専用エンドポイントを追加する（ダイアログ Step1 完了時のみ呼ぶ）

**決定:** B を採用

**理由:**
アドバイスはユーザーが状態スコアを入力した瞬間の `state_score` と `load` を受け取って計算するもの。ポーリングレスポンスに含めると「誰も入力していない時点」でも計算が走り、意味をなさない。また A では `load` を `GET /state` 時点で決定できない。

**docs/ への反映:** 反映済み（`functional-design.md` の API 設計）

## DEC-004 閾値定数の集約先を `session_service.py` に統一

**日付:** 2026-05-09

**背景:**
`ACCUM_DANGER_SEC`（180分）が `timer_service.py` と `session_service.py` の両方に定義される恐れがあった。

**選択肢:**
- A. 各ファイルで個別に定義する
- B. `session_service.py` に集約し、`timer_service.py` はインポートして参照する

**決定:** B を採用

**理由:**
閾値はドメイン判定ロジックであり、`session_service` が責務を持つべき。`timer_service` が定数を持つと、閾値変更時に複数ファイルの変更が必要になり、不整合のリスクがある。

**docs/ への反映:** 反映済み（`functional-design.md` のサービス依存関係図）

## DEC-005 `self` モードはクライアント側で直接起動する（サーバー通知なし）

**日付:** 2026-05-09

**背景:**
`self` モードのダイアログ起動をサーバー経由（`dialog_triggered` フラグ）にするか、クライアント直接起動にするかを決める必要があった。

**選択肢:**
- A. `POST /self-trigger` を追加してサーバー経由でフラグを立て、ポーリングで検知する
- B. 自己申告ボタンクリック時にクライアントが直接 `openDialog('self')` を呼ぶ

**決定:** B を採用

**理由:**
自己申告はユーザーの能動的な操作であり、サーバーを介する必要がない。B はラウンドトリップ不要・コード量が少なく・`dialog/ack` も不要になる。`self` モードはポーリングとは独立して機能する。

**docs/ への反映:** 反映済み（`functional-design.md` のダイアログモード仕様）

## DEC-006 `do_record` の状態整合性（`rest` / `skip` 時のフラグ管理）

**日付:** 2026-05-09

**背景:**
設計レビューで以下の不整合が発見された。
- `action="rest"` 時: `is_breaking = True` にするが `is_running = True` のままだった
- `action="skip"` 時: `is_running = True` にするが `is_breaking` が残留する可能性があった

**決定:**
- `action="rest"`: `is_running = False` を明示的にセットしてから `is_breaking = True`
- `action="skip"`: `is_breaking = False` を明示的にクリアしてから `is_running = True`

**理由:**
`is_running` と `is_breaking` は排他的に動作すべきフラグ。一方が True の間は他方が True になってはならない。バックグラウンドスレッドの `_tick()` は両フラグを独立して評価するため、不整合があると二重カウントが発生する。

**docs/ への反映:** 反映済み（design.md の `do_record` 実装）

## DEC-007 `fetchAdvice()` の `load` パラメータに `prev_load` を使用

**日付:** 2026-05-09

**背景:**
`POST /advice` は `state_score` と `load` を要求するが、`timer`/`self`/`force` モードの Step1 完了時点ではユーザーがまだ `load` を入力していない（Step3 が `load` 入力）。

**選択肢:**
- A. `load` を省略可能にして、省略時はサーバーが `prev_load` を使う
- B. クライアントが最後のポーリング結果（`lastState.prev_load`）を `load` として送る

**決定:** B を採用

**理由:**
A はサーバー側に「省略時のフォールバック」という隠れたロジックが生まれ、テストが難しくなる。B は `prev_load`（前回の作業負荷）がアドバイスの代理指標として合理的であり、クライアントが明示的に送ることで `POST /advice` のインターフェースをシンプルに保てる。

**docs/ への反映:** 不要（実装詳細。`tasklist.md` に明記済み）

## DEC-008 `initCharts()` をフェーズ5（フロントエンド基礎）で実行する

**日付:** 2026-05-09

**背景:**
当初 `initCharts()` はフェーズ7（グラフ・ログ表示）に配置していた。しかしダイアログ内グラフ（`#dialog-state-chart`）はフェーズ6の Step1 で必要になるため、フェーズ6の動作確認時に Chart.js インスタンスが未生成の状態になる問題があった。

**決定:**
`initCharts()`（3インスタンス全て）をフェーズ5に移動する。フェーズ7は `updateCharts()` / `updateDialogChart()` などのデータ更新関数の実装のみとする。

**理由:**
フェーズ完了条件を各フェーズ内で検証できるようにするため。Chart.js インスタンス生成自体は軽量であり、フェーズ5に含めてもコスト増はほぼない。

**docs/ への反映:** 不要（実装順序の判断。`tasklist.md` に反映済み）

## DEC-010 Chart.js のバージョンを CDN で固定する

**日付:** 2026-05-09

**背景:**
CDN URL にバージョン指定がないと、メジャーバージョンアップ時に破壊的変更が適用されてアプリが壊れるリスクがある。

**決定:** Chart.js CDN タグにバージョンをピン留めする（例: `@4.4.0`）。メジャーバージョンは 4.x 固定。

**理由:**
MVP フェーズでは依存ライブラリの自動更新によるリグレッションを防ぐことを優先する。バンドラー不使用の制約（Vanilla JS）のもとでは CDN バージョン固定が唯一の再現性確保手段。

**docs/ への反映:** 反映済み（`architecture.md` の Chart.js バージョン管理方針）

## DEC-011 JS の API レスポンスキーは `snake_case` のまま使用する

**日付:** 2026-05-09

**背景:**
JS の慣習では変数名を `camelCase` にすることが多い。API レスポンスのキー（`session_elapsed` 等）を JS 側で `camelCase` に変換するかどうかを決める必要があった。

**選択肢:**
- A. `const sessionElapsed = state.session_elapsed` のように変換レイヤーを設ける
- B. `state.session_elapsed` のまま使う（変換なし）

**決定:** B を採用

**理由:**
変換レイヤーはコード量が増え、Python 側の命名と JS 側の命名のずれが混乱の原因になる。`snake_case` のまま参照することでバックエンドとフロントエンドの一対一対応が保たれ、デバッグが容易になる。本プロジェクトは単一開発者・ローカル専用のため JS の慣習より一貫性を優先する。

**docs/ への反映:** 反映済み（`development-guidelines.md` の JS コーディング規約）
